Sampling tabulated distributions: build & invert a CDF

14_tabulated_cdf_sampling.cpp — the distributions a renderer most wants to sample have no formula at all. They arrive as a table of numbers. Here is the standard machinery: normalize, prefix-sum into a CDF, and invert it with a binary search.

Example 05 inverted a CDF that had a clean closed-form inverse — $\sqrt U$, $-\ln(1-U)/\lambda$. But the most important distributions in rendering have no formula: the luminance of an HDR environment map, a measured BRDF, the power of each light in the scene. They are just tables of numbers. This file shows how to sample any such table proportionally to its weights, in $O(\log N)$ per sample with a single, branch-coherent binary search — the exact pattern behind environment-map importance sampling and light selection.

One-sentence summary. Treat the table as a piecewise-constant pdf, normalize it, and prefix-sum it into a cumulative table $F$ with $N{+}1$ entries ($F_0=0,\ F_N=1$); to draw a sample, take $u\sim U[0,1)$ and binary-search for the bin with $F_i\le u<F_{i+1}$ — exactly one search per sample, so every GPU lane does the same work — then either interpolate inside the bin (continuous) or just return the index (discrete "which light?").

1. The problem: distributions that are only a table

Inversion and rejection (example 05) both assumed you could write the density down. Real rendering data can't be. An environment map is a grid of measured radiances; the "density" you want to sample is its per-texel luminance — a few million numbers with no analytic form. Same for the list of light powers in a scene, or a tabulated BRDF. The question is always the same: given a table of non-negative weights, draw an index (or a position) with probability proportional to the weights.

Our running test case mirrors the C++ file: a 1-D "sky row" of $W=32$ bins, all dim except three in the middle (the sun) that are $100\times$ brighter. A correct sampler must send almost all of its samples into those three bins.

2. The recipe: normalize, prefix-sum, invert

Treat the table as a piecewise-constant pdf over $N$ unit-width bins. Three steps:

(1) Normalize so the masses sum to one:

$$p_i=\frac{w_i}{\sum_{j}w_j},\qquad \sum_{i=0}^{N-1}p_i=1.$$

(2) Build the CDF as a prefix sum. The cumulative table has $N{+}1$ entries so it holds both edges of every bin:

$$F_0=0,\qquad F_{i+1}=F_i+p_i,\qquad F_N=1.$$

(3) Invert. Draw $u\sim U[0,1)$ and find the bin whose cumulative interval contains it:

$$\text{find }i\ \text{such that}\quad F_i\le u<F_{i+1}.$$

That is the whole idea: $u$ falls into bin $i$ with probability exactly $F_{i+1}-F_i=p_i$, because $u$ is uniform and the interval $[F_i,F_{i+1})$ has length $p_i$. Wide cumulative steps (heavy bins) catch $u$ often; thin steps rarely. The next figure makes the inversion visible.

3. Inverting the CDF, made visual

Plot the CDF $F$ against the domain. Because the pdf is constant within each bin, $F$ rises in a straight ramp across each bin, with slope equal to that bin's probability. To sample, enter at a height $u$ on the left axis, go right until you hit the ramp, and drop down: that landing point is your sample. Steep segments (high $p_i$) span a tall slice of the $u$-axis, so they swallow most draws — that is importance sampling (example 06) falling out of the geometry.

4. Why binary search: one branch-coherent step per sample

For a big table you don't scan linearly — you binary-search the cumulative array for the bin containing $u$. Keep a bracket $[\text{lo},\text{hi}]$ over the edges, look at the middle edge, and discard the half that can't contain $u$:

The cost is $O(\log N)$, but the property that matters on a GPU is subtler: a binary search does exactly the same number of steps for every $u$. Contrast rejection sampling (example 05), whose try-count is random and differs per sample — on a SIMD warp the lucky lanes must idle until the unlucky one finishes, so the whole warp runs at worst-case speed. The fixed, data-independent work of a binary search keeps all 32 lanes in lockstep. That predictability is why this, not rejection, is the workhorse for tables.

5. Continuous vs. discrete sampling

Once you know the bin, there are two things you might want:

Continuous — a real-valued position, e.g. a sub-texel coordinate on the environment map. Interpolate within the bin using how far $u$ reached across its cumulative step:

$$t=\frac{u-F_i}{F_{i+1}-F_i},\qquad x=i+t\ \in[i,i+1).$$

The density at the sample is just the bin's mass — here is why. Conditioned on landing in bin $i$ (probability $p_i$), $u$ is uniform on $[F_i,F_{i+1})$, and $t$ rescales that interval linearly onto $[0,1)$, so $x$ is uniform across the unit-wide bin. Its density is therefore $\Pr[\text{bin }i]$ times the uniform density on a unit interval, $p_i\times 1$:

$$p(x)=F_{i+1}-F_i=p_i,\qquad x\in[i,i+1),$$

the value MIS (example 09) needs as the pdf of this strategy. (Sampling confirms it: within a sun bin the sub-positions come out flat to within noise.)

Discrete — just the index $i$. "Which of these $N$ lights do I shade?" is answered by the same prefix-sum CDF and the same binary search; you return the bin number and stop. No interpolation, no density-within-a-bin — the probability of picking light $i$ is exactly $p_i$.

6. Verifying it: the sky row, sampled

Draw samples through the inverse CDF and histogram where they land. The empirical frequencies must converge to the pdf — most of them piling into the three sun bins. The C++ file draws a million samples; do the same here in batches and watch the bars settle.

7. The discrete case: picking a light by power

The "which light?" problem is the discrete sampler verbatim. Three lights with powers $90:1:9$ become probabilities $0.90,0.01,0.09$; their CDF cuts $[0,1)$ into three intervals, and a uniform $u$ picks whichever it falls in. The tiny middle interval (width $0.01$) is hit about once in a hundred — proportional to its power, no more.

8. Where this lives in the renderer

This single routine is everywhere a renderer faces measured data it can only tabulate:

  • Environment-map importance sampling. Build a pdf from per-texel luminance and sample directions toward the bright parts of the sky instead of firing rays uniformly — the 1-D version here, extended to 2-D in example 15 (sample a row by its total power, then a texel within that row).
  • Light selection. The discrete pick of §7 chooses which light to sample in a many-light scene, weighted by power (or a better importance estimate).
  • Any measured BRDF or phase function supplied as a table rather than a formula.

The companion PathTracingDemo renderer samples an environment cubemap for its lighting; turning that map's luminance into an importance distribution is exactly the construction on this page. Note the returned density $p_i$ is what lets these strategies be combined with others through MIS (example 09).

9. The code

The four routines are tiny. BuildCdf is the prefix sum (pinning the last entry to exactly $1$ against float drift); FindBin is the branch-coherent binary search; the two samplers differ only in what they return.

// Build a CDF (prefix sum) from a pdf table. Result has pdf.size()+1 entries.
std::vector<float> BuildCdf(const std::vector<float>& pdf) {
    std::vector<float> cdf(pdf.size() + 1, 0.0f);
    for (size_t i = 0; i < pdf.size(); ++i) cdf[i + 1] = cdf[i] + pdf[i];
    cdf.back() = 1.0f;                     // pin the top against float drift
    return cdf;
}

// Find the bin i with cdf[i] <= u < cdf[i+1] by binary search.
int FindBin(const std::vector<float>& cdf, float u) {
    int lo = 0, hi = (int)cdf.size() - 1;          // searching edges [0, N]
    while (hi - lo > 1) {
        int mid = (lo + hi) >> 1;
        if (cdf[mid] <= u) lo = mid; else hi = mid;
    }
    return lo;                                       // in [0, N-1]
}

// Continuous: pick a bin, then interpolate within it. Also returns the density.
float SampleContinuous(const std::vector<float>& cdf, float u, float& outPdf) {
    int i = FindBin(cdf, u);
    float f0 = cdf[i], f1 = cdf[i + 1];
    float t = (f1 > f0) ? (u - f0) / (f1 - f0) : 0.0f;
    outPdf = f1 - f0;                                // bin width is 1 -> density = mass
    return i + t;
}

// Discrete: just the bin index, distributed proportionally to pdf[i].
int SampleDiscrete(const std::vector<float>& cdf, float u) {
    return FindBin(cdf, u);
}
Takeaway. Any non-negative table becomes a sampler in three lines of setup (normalize, prefix-sum) and one $O(\log N)$ binary search per draw. It is exact (no rejection waste), unbiased, returns its own density for MIS, and does identical work on every lane — the reason it underlies environment-map and light sampling in production path tracers.
