Adaptive sampling: spend samples where the variance is

17_adaptive_sampling.cpp — example 13 gave every pixel the same sample count, but the noise was not uniform. With a fixed budget, send more samples to the high-variance pixels and fewer to the calm ones — the Neyman allocation.

In example 13 the speckle lived only along edges; flat interiors converged after a handful of samples. Giving those interiors the same 128 samples as the edges is pure waste. Adaptive sampling reallocates a fixed total budget: estimate each pixel's noisiness, then pour samples into the noisy pixels and starve the calm ones. The result is the same number of rays at a much lower error — here 3.7× lower.

One-sentence summary. To minimize the total error $\sum_i \sigma_i^2/n_i$ under a fixed sample budget $\sum_i n_i = B$, give pixel $i$ a share proportional to its standard deviation — $n_i \propto \sigma_i$ (the Neyman allocation) — which you get by taking a cheap pilot pass everywhere to estimate each $\sigma_i$, then handing out the rest of the budget by that weight; the edges soak it up and the flat interiors keep only a floor.

1. The waste in uniform sampling

A pixel deep inside a flat-shaded region has zero variance: every sub-pixel sample returns the same value, so one sample already nails it and the other 127 add nothing. A pixel straddling an edge has high variance: its samples disagree, and it needs many to average out. Uniform sampling can't tell them apart — it spends equally and so over-pays the interiors and under-pays the edges. The fix is to measure each pixel's variance and budget accordingly.

2. How many samples each pixel deserves

If pixel $i$ has per-sample standard deviation $\sigma_i$ and receives $n_i$ samples, the variance of its averaged estimate is $\sigma_i^2/n_i$. We want to choose the $n_i$ to minimize the total error

$$\min_{\{n_i\}}\ \sum_i \frac{\sigma_i^2}{n_i}\quad\text{subject to}\quad \sum_i n_i = B.$$

A Lagrange multiplier turns this into $-\sigma_i^2/n_i^2 + \lambda = 0$ for every pixel, i.e. $n_i = \sigma_i/\sqrt\lambda$. The constant drops out under the budget constraint, leaving the classic Neyman allocation:

$$\boxed{\,n_i \;\propto\; \sigma_i\,}\qquad\Longleftrightarrow\qquad \frac{n_i}{n_j}=\frac{\sigma_i}{\sigma_j}.$$

Allocate in proportion to the standard deviation, not the variance. The toy below has just two pixels — a noisy one ($\sigma_A$) and a calm one ($\sigma_B$) sharing a budget of 200 samples. Slide the split and watch the total error; it bottoms out exactly where $n_A/n_B=\sigma_A/\sigma_B$.

3. Estimating $\sigma_i$: the pilot pass

Neyman needs each $\sigma_i$, which we don't know in advance — so we measure it. A cheap pilot pass draws a few samples (here 32) at every pixel and computes the running sample standard deviation (Welford, example 01). Then the remaining budget is dealt out by the weight $\sigma_i + \varepsilon$, and the pilot samples are reused rather than thrown away.

Why the floor $\varepsilon$. The pilot's $\hat\sigma_i$ is itself noisy. A thin-edge pixel can unluckily draw 32 nearly-identical pilot samples and look flat ($\hat\sigma_i\approx 0$), which would starve it of all further samples. Adding a small floor ($\varepsilon=0.01$) to the weight guarantees every pixel keeps a baseline allocation, so a mis-estimated pixel is never dropped below the uniform level.

4. Where the budget actually goes

Run the pilot on example 13's scene and turn each pixel's $\sigma_i$ into a sample-density map. It is bright exactly along the circle edges — the only places with variance — and dark across every flat interior. In the C++ run the busiest edge pixel receives 4378 samples while a flat interior keeps just the 32-sample pilot floor: a 130× spread, all decided by the pilot's $\sigma_i$.

Render it live below at a matched budget (a smaller $120\times120$ scene so it runs in the browser); the error maps (estimate − reference, amplified) make it visible: the uniform map glows with edge noise the adaptive map has scrubbed away, at no extra cost.

Why it must help — and by how much. Plug the Neyman optimum $n_i=B\,\sigma_i/\sum_j\sigma_j$ back into the error $\sum_i\sigma_i^2/n_i$ and it collapses to a clean optimum, set against the uniform allocation $n_i=B/M$:

$$E_{\text{Neyman}}=\frac{\big(\sum_i\sigma_i\big)^2}{B},\qquad\qquad E_{\text{unif}}=\frac{M}{B}\sum_i\sigma_i^2 .$$

Their ratio $E_{\text{unif}}/E_{\text{Neyman}}=M\sum_i\sigma_i^2\big/\big(\sum_i\sigma_i\big)^2$ is $\ge 1$ by Cauchy–Schwarz — adaptive is never worse than uniform — with equality only when every $\sigma_i$ is identical. The win grows with how unequal the $\sigma_i$ are, and here they are extreme: ${\approx}97.7\%$ of pixels have $\sigma_i\approx0$ and a $2.3\%$ edge sliver carries all the noise. That dispersion puts the idealized ceiling at $\sqrt{M\sum\sigma^2/(\sum\sigma)^2}\approx7.7\times$ in RMS; the measured $3.72\times$ is lower because the $\varepsilon$-floor and the 32-sample pilot still spend real budget on the already-converged flats.

6. In the renderer

Production renderers do exactly this, usually with a perceptual error metric rather than raw variance: render a pilot, measure where the image is still noisy, and keep tracing only there until each region hits an error target. It is one of the highest-leverage knobs in a renderer because noise is so unevenly distributed — caustics, penumbrae, and glossy highlights cost orders of magnitude more than flat walls.

The PathTracingDemo renderer does the simple thing instead: every pixel gets the same progressive average each frame (the uniform baseline of example 13), so it keeps spending on long-converged regions while the hard pixels lag. Driving its sample budget from a per-pixel variance estimate — the pilot-and-Neyman idea here — is the kind of upgrade that would buy a cleaner image for the same frame time.

7. The code

The pilot pass produces each pixel's $\sigma_i$; the allocation loop turns the floored weights into per-pixel counts that sum to the budget, never dropping below the pilot.

// pilot pass: a few samples everywhere -> per-pixel sigma -> Neyman weight
for (int i = 0; i < kNumPix; ++i) {
    double m2; pilotMean[i] = SampleMean(i % W, i / W, kPilot, 2u, &m2);
    double sigma = std::sqrt(m2 / (kPilot - 1));   // sample std dev (Welford M2)
    weight[i] = sigma + kSigmaFloor;               // floor: never starve a pixel
    wSum += weight[i];
}

// allocate the whole budget in proportion to weight (n_i ~ sigma_i)
for (int i = 0; i < kNumPix; ++i) {
    int n = (int)std::llround(kBudget * weight[i] / wSum);
    if (n < kPilot) n = kPilot;                     // reuse the pilot work
    int extra = n - kPilot;
    double extraMean = extra > 0 ? SampleMean(i % W, i / W, extra, 3u) : 0.0;
    adapt[i] = (kPilot * pilotMean[i] + extra * extraMean) / n;   // combine
}
Takeaway. Noise is not spread evenly across an image, so neither should your samples be. Measure each pixel's variance with a cheap pilot pass and allocate the rest of a fixed budget in proportion to its standard deviation ($n_i\propto\sigma_i$, Neyman): the edges get hundreds of times more samples than the flats, and the whole image converges several times faster for the same ray count.
