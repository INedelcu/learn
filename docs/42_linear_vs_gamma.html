Average in linear space: the gamma trap

42_linear_vs_gamma.cpp — a monitor's 8-bit values are sRGB-encoded, not proportional to light. Averaging those encoded values instead of the linear radiance behind them is one of the most common and most visible rendering bugs — the midtones always come out too dark.

Monte Carlo rendering is an averaging machine: it estimates each pixel by averaging the radiance of many paths. That average is only correct if it happens in linear radiance. But the numbers a monitor consumes are sRGB-encoded — roughly $\text{value}\approx L^{1/2.2}$ — and encoding is nonlinear, so the average of encoded values is not the encoding of the average. Blend, downsample, or anti-alias in display space and you get a specific, always-in-the-same-direction error: the midtones darken. This is exactly why the path tracer accumulates linear radiance and applies the sRGB curve only at the final blit.

One-sentence summary. The sRGB encode $f$ is concave with $f(0)=0,\ f(1)=1$, so by Jensen's inequality $f$ lies above the diagonal on $[0,1]$; encoding-then-averaging gives the right answer $f(\text{cov})$ while averaging-then-displaying gives the too-low $\text{cov}$, and the gap $f(\text{cov})-\text{cov}\ge0$ is always a darkening — peaking near $60$–$73$ of $255$ levels in the midtones (e.g. a $50\%$ edge should display as $188/255$ but comes out $128/255$).

sRGB encode $f(c)$ correct: encode the linear average wrong: average the encoded values

1. Monitors are nonlinear

An 8-bit pixel value is not proportional to the light the monitor emits. The sRGB standard defines the encoding from linear radiance $c\in[0,1]$ to stored value:

$$f(c)=\begin{cases}12.92\,c & c\le 0.0031308,\\[2pt] 1.055\,c^{1/2.4}-0.055 & \text{otherwise,}\end{cases}$$

a tiny linear toe near black grafted onto a power curve with exponent $1/2.4$ (the whole thing approximates $c^{1/2.2}$). It is anchored at the corners, $f(0)=0$ and $f(1)=1$, and it is concave: $f''<0$ on the power segment. Concavity is the entire story. It means the curve bows above the straight line joining its endpoints — so $f(c)\ge c$ for every $c\in[0,1]$, with equality only at $0$ and $1$. Mid-gray $c=\tfrac12$ encodes not to $0.5$ but to $f(0.5)=1.055\cdot0.5^{1/2.4}-0.055=0.735$, i.e. $188/255$. The monitor needs a high code to make middling light.

2. The trap: averaging encoded values

Take the canonical case: a pixel whose area is fraction $\text{cov}$ covered by a white surface ($L=1$) and the rest by black ($L=0$) — an anti-aliased edge. The true light reaching the eye is the linear average

$$L=\text{cov}\cdot 1+(1-\text{cov})\cdot 0=\text{cov},\qquad\text{correct display}=f(\text{cov}).$$

Now the bug. If instead you average the two surfaces' encoded pixel values — which is what happens when you blend, downsample, or filter in display space — you compute

$$\text{cov}\cdot f(1)+(1-\text{cov})\cdot f(0)=\text{cov}\cdot 1+(1-\text{cov})\cdot 0=\text{cov},$$

and then send $\text{cov}$ straight to the display as if it were already a valid code. The two answers are $f(\text{cov})$ versus $\text{cov}$, and since $f(\text{cov})\ge\text{cov}$ the wrong one is always darker or equal, never brighter. This is just Jensen's inequality for a concave $f$: $f(\mathbb E[L])\ge \mathbb E[f(L)]$, with the endpoints black and white making both sides collapse to the clean forms above. The error vanishes at the endpoints (pure black and pure white encode the same either way) and bulges in between. Its maximum sits in the low midtones: differentiating $f(c)-c$ and setting $f'(c)=1$ gives $c\approx0.22$, right where the table peaks at $73$ levels. That is not a subtle tint — $73/255$ is nearly a third of the entire tonal range, wrong, on every anti-aliased edge in the frame.

3. The 50% edge, exactly

The half-covered edge is worth doing by hand, because it is the one everyone pictures. Correct:

$$f(0.5)=1.055\cdot 0.5^{1/2.4}-0.055=1.055\cdot0.7492-0.055=0.7354\ \Rightarrow\ \lfloor 0.7354\cdot255\rceil=188.$$

Wrong: average the codes $\{0,255\}$ to get $128$. The half-and-half edge that should read as a bright $188/255$ gray instead renders as a murky $128/255$ — a $60$-level hole punched into every $50\%$ edge in the image. And it compounds: mip-map generation, texture downsampling, and alpha compositing done in sRGB all make the same mistake, which is why a naively-scaled thumbnail looks darker than the original.

4. Why the renderer accumulates in linear

Every averaging step in the pipeline must live in linear space, and the encode must happen exactly once, last. This demo's accumulator is built that way: radiance is stored in a floating-point R32G32B32A32_SFloat buffer and each new sample is folded in with the progressive average lerp(prev, new, 1/(step+1)) on linear values — the same running mean the whole series uses. Only at the final blit to the back buffer is the sRGB curve applied. Because the buffer is float and linear, there is no encoding sitting between samples to corrupt the average: the thousands of paths accumulated per pixel, the reconstruction filter of example 41, and any downsampling all compose correctly, and the nonlinearity is deferred to a single well-defined output transform. Mixing the two spaces anywhere upstream is the bug this example exists to inoculate against.

5. The code

The complete sample on GitHub: 42_linear_vs_gamma.cpp.

The sRGB encode, and the one-line difference between the correct and the wrong pixel.

double ToSRGB(double c) {                    // linear radiance -> display code, concave
    c = c < 0 ? 0 : c > 1 ? 1 : c;
    return c <= 0.0031308 ? 12.92 * c
                          : 1.055 * std::pow(c, 1.0/2.4) - 0.055;
}

// A pixel that is fraction 'cov' white (L=1) over black (L=0):
int correct = (int)std::lround(ToSRGB(cov) * 255);   // encode(linear average)  -> right
int wrong   = (int)std::lround(cov * 255);           // average of encoded {0,255} -> too dark
// correct - wrong >= 0 always: f is concave, so it bows above the diagonal (Jensen)
Takeaway. The sRGB encode $f$ is concave with $f(0)=0,f(1)=1$, so it lies above the diagonal and $f(c)\ge c$ everywhere. Averaging linear radiance and encoding once gives the correct pixel $f(\text{cov})$; averaging already-encoded values gives $\text{cov}$, which is always darker — up to $73/255$ in the low midtones, $60/255$ at a $50\%$ edge ($188$ vs $128$). Monte Carlo rendering is averaging, so keep every accumulation, blend, and filter in linear radiance and apply the sRGB curve only when writing the final pixel. Mixing the two spaces is one of the most common and most visible rendering bugs.
