Per-pixel RNG seeding: why decorrelation matters

36_rng_decorrelation.cpp — every pixel gets its own random stream, and how you seed it is easy to get wrong. A poorly mixed seed makes neighbouring pixels draw correlated numbers — visible structure no amount of averaging removes cleanly.

A path tracer gives every pixel its own random number stream. Seeding those streams sounds trivial, but a careless choice — a raw linear index, an under-mixed hash — makes adjacent pixels draw nearly the same numbers. The result isn't "more" or "less" random; each pixel's value is still uniform. It's that neighbours are correlated, so the error has low-frequency structure — diagonal banding — that the eye latches onto and averaging cannot easily clean up. The fix is to hash the seed so nearby pixels scatter to unrelated states.

One-sentence summary. Seed each pixel's stream by hashing its coordinates (and frame, and convergence step); a good hash sends pixel $i$ and pixel $i{+}1$ to totally unrelated states, so their samples are independent and the per-pixel noise is structureless — whereas a smooth/under-mixed seed leaves neighbours correlated, producing visible banding. The test: the mean absolute difference of adjacent pixels' first samples is $\tfrac13$ for independent uniforms and far below it when they are correlated.

bad seed (smooth / under-mixed) good seed (Wang hash)

1. Each pixel's stream needs its own seed

To make each pixel reproducible and parallel-safe, a renderer derives that pixel's random stream from a seed built out of its identity — pixel coordinates, the frame index, the convergence step. Pixels are laid out on a grid, so their raw seeds are almost identical: pixel $(x,y)$ and $(x{+}1,y)$ differ by one. If the seed-to-stream map is smooth — barely changing for a one-unit change in input — then adjacent pixels produce nearly the same samples, and the image fills with structure instead of noise. Whatever produces the stream, the seed must be scrambled first.

2. Bad seed vs good seed

Fill an image where each pixel's value is its first random sample, two ways. Bad: a smooth function of position, $\operatorname{frac}(0.123x+0.317y)$ — what "low-effort" seeding gives you, adjacent pixels barely differ. Good: a Wang hash of the pixel index, which scrambles nearby seeds into unrelated values.

The $\tfrac13$ test. A clean number for "are neighbours independent?" is the mean absolute difference of adjacent pixels. For two independent uniforms $X,Y\sim U[0,1]$,

$$\mathbb E\,|X-Y|=\int_0^1\!\!\int_0^1|x-y|\,dx\,dy=2\!\int_0^1\!\!\int_0^y(y-x)\,dx\,dy=2\!\int_0^1\frac{y^2}{2}\,dy=\frac13.$$

So independent neighbours average $\tfrac13\approx0.333$; a much smaller value means correlation. The C++ run measures the bad scheme at $0.2157$ (far below $\tfrac13$ — correlated) and the Wang-hashed scheme at $0.3289$ ($\approx\tfrac13$ — independent).

3. It's correlation, not randomness

The bad scheme is not "less random" — every pixel's value is still a perfectly good uniform in $[0,1]$. What's wrong is the joint distribution: neighbours move together. 4. In the renderer

The PathTracingDemo renderer seeds its generator by mixing the pixel coordinates, the frame index, and the convergence step through a Wang hash (WangHash / RandomFloat01 in Utils.hlsl). Folding in the frame and step matters as much as the pixel: it decorrelates a pixel's stream across time too, so successive frames add fresh independent samples rather than re-drawing the same ones. This is the unglamorous plumbing under every sample the renderer takes — and it interacts with everything else here: Owen scrambling (example 35) is a fancier per-pixel decorrelation, and the goal of blue-noise seed masks (example 30) is to make even the residual correlation high-frequency. Good seeding is as important as the sampling math; a great estimator on a correlated stream still looks broken.

5. The code

The complete sample on GitHub: 36_rng_decorrelation.cpp.

The Wang hash turns a near-sequential index into a decorrelated state.

// Wang hash: thoroughly scrambles an integer seed (the renderer's family).
uint32_t WangHash(uint32_t s) {
    s = (s ^ 61u) ^ (s >> 16);
    s *= 9u;
    s = s ^ (s >> 4);
    s *= 0x27d4eb2du;
    s = s ^ (s >> 15);
    return s;
}
uint32_t idx = y * W + x;
float bad  = std::fmod(0.123f*x + 0.317f*y, 1.0f);   // smooth seed -> correlated neighbours
float good = WangHash(idx) * 0x1p-32f;               // hashed seed -> independent neighbours
// mean |adjacent diff|:  bad 0.216 (correlated)  vs  good 0.329 (~1/3, independent)
// the renderer hashes (pixel, frame, convergence step) so streams decorrelate in space AND time.
Takeaway. Per-pixel RNG streams must be seeded through a hash, or adjacent pixels — whose raw seeds differ by one — draw correlated numbers and the image shows banding that averaging can't remove. The mean adjacent-pixel difference is $\tfrac13$ for independent uniforms; a smooth seed measures $0.22$ (correlated), a Wang hash $0.33$. It is not about being "more random" but about decorrelation, and hashing the (pixel, frame, step) seed — as the renderer does — is what turns per-pixel streams into clean, structureless noise.
