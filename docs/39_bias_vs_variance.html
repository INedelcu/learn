Bias vs variance: unbiased, consistent, and the difference

39_bias_vs_variance.cpp — variance is noise that averages away as 1/N; bias is a wrong answer that doesn't. Path tracing is unbiased; photon mapping and denoisers are biased-but-consistent — they need a smoothing parameter driven to zero, or more samples just nail down the blur.

Two different things can be wrong with an estimator. Variance is scatter around its average — noise, which shrinks as $1/N$. Bias is when that average is the wrong number — and it does not shrink with $N$ at all. Path tracing is unbiased: its expected value is the true integral, so more samples drive the error to zero. Many faster methods — photon mapping, irradiance caching, denoisers — are biased but consistent: they reach the truth only if a smoothing parameter is also driven to zero as $N$ grows. Hold that parameter fixed and you converge precisely to the wrong, blurred answer.

One-sentence summary. A kernel density estimate — photon mapping's exact math — counts samples within a half-width $h$ of $x_0$; its expectation is the window average $p(x_0)+\tfrac16 p''(x_0)h^2+\dots$, so a fixed $h$ leaves a permanent $O(h^2)$ bias no matter how large $N$ is (precise but wrong), while shrinking $h\to0$ with $N$ removes the bias (consistent) at the price of convergence slower than the unbiased $1/\sqrt N$.

1. Two ways to be wrong

Split an estimator's error into two pieces. The mean-squared error decomposes exactly as

$$\operatorname{MSE}=\underbrace{\big(\mathbb E[\hat I]-I\big)^2}_{\text{bias}^2}+\underbrace{\operatorname{Var}[\hat I]}_{\text{variance}}.$$

Variance is the part that averages out: it falls as $1/N$ (example 04), so with enough samples the noise vanishes. Bias is the part that doesn't: if $\mathbb E[\hat I]\ne I$, no number of samples fixes it — you just measure the wrong value more precisely. An estimator is unbiased if $\mathbb E[\hat I]=I$ (path tracing, every estimator in this series so far), and consistent if $\hat I\to I$ as $N\to\infty$. Unbiased implies consistent, but a biased estimator can still be consistent — if its bias shrinks with $N$.

2. Kernel density estimation and the $h^2$ bias

The cleanest example — and exactly photon mapping's math — is estimating a density $p$ at a point $x_0$ by counting samples $X_i\sim p$ within a kernel of half-width $h$:

$$\hat p(x_0)=\frac{\#\{\,i:|X_i-x_0|<h\,\}}{2hN}.$$

Its expectation is not $p(x_0)$ but the average of $p$ over the window. Taylor-expanding $p$ around $x_0$ and integrating (the odd linear term cancels):

$$\mathbb E[\hat p(x_0)]=\frac{1}{2h}\int_{x_0-h}^{x_0+h}\! p(x)\,dx =\frac{1}{2h}\Big(2h\,p(x_0)+\tfrac13 p''(x_0)h^3+\cdots\Big)=p(x_0)+\frac{p''(x_0)}{6}h^2+\cdots$$

The bias is $\tfrac16 p''(x_0)h^2$ — proportional to $h^2$ and the curvature, and independent of $N$. For $p(x)=3x^2$ ($p''=6$) it is exactly $h^2$. The fixed-$h$ estimate gets ever more precise — its variance shrinks — but it converges to $0.75+h^2=0.7525$, not $0.75$: a permanent bias of $h^2=0.0025$ that more samples cannot remove. The shrinking-$h$ estimate is consistent — its bias melts toward $0$. But that costs convergence speed: with variance $\sim1/(Nh)$ and bias $\sim h^2$, the MSE $\sim h^4+1/(Nh)$ is minimized at $h\propto N^{-1/5}$ (hence the $N^{-1/5}$ schedule), giving RMSE $\propto N^{-2/5}$ — strictly slower than the unbiased $N^{-1/2}$. 4. In the renderer

This is the fault line between two families of renderers. Path tracing (this whole series, and the PathTracingDemo renderer) is unbiased: it is noisy, but every extra sample provably moves the image toward ground truth, and the noise is the honest $1/\sqrt N$. Photon mapping, irradiance caching, and radiance caching are biased: they blur illumination over a kernel (the photon-gather radius, the cache spacing) to trade noise for speed, and are only consistent if that radius shrinks as photons are added. Denoisers are the extreme case — heavily biased, with the "smoothing parameter" being the filter strength; they look clean but can be flatly wrong in ways more samples don't fix unless the filter relaxes. The practical rule: unbiased methods let you trust convergence and error bars (example 04); biased methods buy early speed but you must know what smoothing you dialed in, and whether it goes to zero.

5. The code

The complete sample on GitHub: 39_bias_vs_variance.cpp.

A box-kernel density estimate at two bandwidths — one fixed, one shrinking with $N$.

double hFix = 0.05;                          // fixed bandwidth  -> permanent h^2 bias
double hShr = 0.4 * std::pow((double)N, -0.2);   // shrinking (MSE-optimal order N^-1/5)

long long cntFix = 0, cntShr = 0;
for (long long i = 0; i < N; ++i) {
    double X = std::pow(rng.NextFloat(), 1.0/3.0);          // X ~ p(x) = 3x^2
    if (std::fabs(X - x0) < hFix) ++cntFix;
    if (std::fabs(X - x0) < hShr) ++cntShr;
}
double estFix = cntFix / (2.0 * hFix * N);   // E = p(x0) + p''(x0)/6 * h^2  -> biased by h^2
double estShr = cntShr / (2.0 * hShr * N);   // bias -> 0 as h -> 0, but RMSE ~ N^-2/5 (slower)
Takeaway. Error is bias plus variance: variance is noise that averages away as $1/N$; bias is a wrong mean that doesn't. A kernel/photon estimate is biased by $\tfrac16 p''(x_0)h^2$ — a fixed kernel converges precisely to the wrong, blurred value, while a kernel shrinking as $h\propto N^{-1/5}$ is consistent but converges as $N^{-2/5}$, slower than unbiased path tracing's $N^{-1/2}$. Unbiased methods trade speed for trustworthy convergence; biased ones (photon mapping, caching, denoisers) trade correctness for speed, and are only right if their smoothing goes to zero.
