Reconstruction filters: turning samples into a pixel

41_reconstruction_filters.cpp — a pixel is not the plain average of the samples that fell in it. It is a weighted average defined by a reconstruction filter centered on the pixel, and the shape of that filter is the last sharpness-vs-aliasing knob on the image.

Path tracing produces a cloud of samples at continuous positions on the film; the display wants one value per pixel. The bridge is a reconstruction filter $k$ centered on each pixel: the pixel value is the filter-weighted average of nearby samples. The choice of $k$ is not cosmetic — a box filter is sharp but aliases, a Gaussian is smooth but blurry, and the Mitchell–Netravali filter buys apparent sharpness with small negative lobes that make edges ring. Every property that matters — brightness preservation, the softness of an edge, the overshoot of ringing — falls straight out of two integrals of $k$.

One-sentence summary. A filter must integrate to $1$ or it rescales image brightness (box, tent, Gaussian all check out at $1.0000$); reconstructing a step edge is just the filter's running integral, which is why every filter passes through exactly $0.5$ at the edge center and why the box gives a hard 1-pixel ramp — and a filter with negative lobes (Mitchell) has a non-monotone running integral, so the edge overshoots past $1$ and dips below $0$: that overshoot is ringing, and it is the very thing that makes the filter look sharp.

1. A pixel is a filter-weighted average

A pixel centered at $p$ does not simply average the samples that landed inside its square. It takes a weighted average, with weights given by a reconstruction filter $k$ centered on $p$:

$$c(p)=\frac{\sum_i k(x_i-p)\,c_i}{\sum_i k(x_i-p)},$$

where $x_i$ are sample positions and $c_i$ their colors. A wide filter reaches into neighboring pixels; a narrow one ignores them. The box filter — flat over $|x|<\tfrac12$ — recovers the naïve "average the samples in this pixel". The others fall off with distance, so a sample near the pixel center counts more than one near the edge. The four classic 1D kernels (renderers use the 2D product $k(x)k(y)$):

2. A filter must integrate to one

Feed the pipeline a flat field of constant radiance $c_0$. Every reconstructed pixel should come back as $c_0$ — no brightening, no darkening. In the continuous limit that says $\int k(x)\,dx=1$: the weights must sum to unity. A kernel that integrates to $1.05$ would make a uniform image $5\%$ too bright. The first three check out by hand:

$$\int_{-1/2}^{1/2}\!\!1\,dx=1,\qquad \int_{-1}^{1}(1-|x|)\,dx=2\!\int_0^1(1-x)\,dx=2\cdot\tfrac12=1,\qquad \frac{1}{\sigma\sqrt{2\pi}}\!\int_{\mathbb R}\! e^{-x^2/2\sigma^2}dx=1,$$

the Gaussian using $\sigma=\tfrac12$, whose normalizer $1/(\sigma\sqrt{2\pi})=0.79788$ is exactly the constant in the code. The Mitchell–Netravali cubic is normalized by construction for every $(B,C)$ — the $/6$ in its definition is precisely that normalizer. The Gaussian's $0.9999$ is not error in the formula — it is the truncation: a true Gaussian has infinite support, and clipping it at radius $2$ discards the $0.01\%$ of mass in the tails. In practice you either accept it or renormalize by the truncated sum.

3. Reconstructing an edge is the filter's running integral

The sharpest test signal is a step edge: black for $t<0$, white for $t\ge 0$, i.e. $s(t)=[t\ge0]$. Reconstructing it at a pixel centered at $p$ convolves the step with the filter:

$$R(p)=\int s(t)\,k(t-p)\,dt=\int_{0}^{\infty} k(t-p)\,dt=\int_{-p}^{\infty} k(u)\,du = 1-C(-p),$$

substituting $u=t-p$ and writing $C(x)=\int_{-\infty}^{x}k$ for the filter's cumulative. So the reconstructed edge is nothing but the filter's running integral, read backwards. Two consequences drop out immediately. First, at the edge center $p=0$ every normalized symmetric filter gives $R(0)=\int_0^\infty k = \tfrac12$ — half the mass sits on each side. The C++ table shows all four columns hitting exactly $0.5000$ at $p=0$, a clean check on the whole pipeline. Second, for the box, $R(p)=\int_{-p}^{1/2}1\,du=\tfrac12+p$ clamped to $[0,1]$: a straight ramp exactly one pixel wide. Wider filters spread the ramp wider and softer.

4. Ringing is exactly the negative lobes

Whether an edge can overshoot is decided entirely by the sign of $k$. If $k\ge0$ everywhere — box, tent, Gaussian — then $C(x)=\int_{-\infty}^x k$ climbs monotonically from $0$ to $1$, so $R(p)=1-C(-p)$ is trapped in $[0,1]$. No overshoot is possible; the price is that a non-negative kernel wide enough to suppress aliasing must also blur. The Mitchell cubic breaks this on purpose. On $1<|x|<2$ its outer lobe goes negative — plugging $B=C=\tfrac13$ into the outer piece gives

$$k(x)=\frac{-\tfrac73 x^3+12x^2-20x+\tfrac{32}{3}}{6},\qquad k(1.5)=\frac{-0.208}{6}\approx-0.035 < 0.$$

A negative lobe makes $C$ non-monotone: near $x=-2$ the cumulative first dips below $0$, so $R(p)=1-C(-p)$ climbs above $1$ for $p\in(1,2)$; symmetrically it dips below $0$ for $p\in(-2,-1)$. That is ringing — a bright fringe on the light side of an edge, a dark fringe on the dark side. The C++ table shows it to the digit:

Only Mitchell breaks the bounds, and that little $\pm1.6\%$ overshoot at $p=\pm1.25$ is what the eye reads as a crisp, sharp edge. Sharpness and ringing are the same phenomenon.

5. Choosing a filter, and what the renderer does

The filter is the last aliasing-vs-sharpness knob on the image. Box: sharpest passband but leaks aliasing badly — jaggies and moiré. Tent: cheap, mild smoothing, a common default when a real filter is overkill. Gaussian: no ringing ever (it is strictly positive), at the cost of visible blur. Mitchell–Netravali with $B=C=\tfrac13$: the tuned sweet spot most renderers ship as default, sharp-looking with only faint ringing.

This demo takes the simplest choice by construction. The ray generator jitters each primary ray uniformly within the pixel square and accumulates the results with equal weight — a uniform jitter over the pixel is exactly a box reconstruction. That is the sibling of example 12: to use a Gaussian filter instead, you would importance-sample the filter — jitter primary rays with a Gaussian offset (Box–Muller) rather than a uniform one — so the sample density itself carries the filter weight and no explicit weighted sum is needed. Choosing the reconstruction filter and choosing the sample jitter distribution are two views of the same decision.

6. The code

The four kernels, and the Mitchell cubic whose negative outer lobe produces the ringing.

float Box(float x)  { return std::fabs(x) < 0.5f ? 1.0f : 0.0f; }        // flat, radius 1/2
float Tent(float x) { x = std::fabs(x); return x < 1.0f ? 1.0f - x : 0.0f; }
float Gauss(float x){ return std::fabs(x) > 2.0f ? 0.0f
                          : 0.7978846f * std::exp(-2.0f*x*x); }          // sigma=1/2 -> integral ~1

float Mitchell(float x) {               // B = C = 1/3, radius 2, normalized by the /6
    const float B = 1.0f/3, C = 1.0f/3;
    x = std::fabs(x); float x2 = x*x, x3 = x2*x;
    if (x < 1) return ((12-9*B-6*C)*x3 + (-18+12*B+6*C)*x2 + (6-2*B)) / 6.0f;
    if (x < 2) return ((-B-6*C)*x3 + (6*B+30*C)*x2 + (-12*B-48*C)*x + (8*B+24*C)) / 6.0f;
    return 0.0f;                         // outer piece goes NEGATIVE near x=1.5 -> ringing
}

// Reconstruct a step edge s(t)=[t>=0] at center p:  integral s(t) k(t-p) dt / norm.
double R = 0, dx = 2*radius / M;
for (int i = 0; i < M; ++i) {
    double t = p - radius + (i + 0.5) * dx;
    if (t >= 0) R += k(t - p);           // the step is 1 only for t >= 0
}
R = R * dx / norm;                       // = 1 - C(-p): the filter's running integral
Takeaway. A pixel is a filter-weighted average of samples, and everything the filter does is two integrals. It must integrate to $1$ (box/tent/gauss/mitchell $=1.0000/1.0000/0.9999/1.0000$) or it rescales brightness. Reconstructing an edge is the filter's running integral $R(p)=1-C(-p)$, so all symmetric filters pass through $0.5$ at the edge and the box gives a one-pixel ramp. A non-negative filter can never overshoot but must blur; Mitchell's negative lobes make its running integral non-monotone, so its edge rings ($1.0161$ at $p=1.25$) — and that overshoot is exactly what reads as sharpness. Filter choice and jitter distribution (example 12) are the same knob.
